### **Ground Rules for Agent Evolution**

1. **Feature Inheritance:** The next-generation agent (Agent[N+1]) must incorporate all features of the current agent.
2. **Issue Resolution:** Agent[N+1] should aim to resolve or mitigate any existing issues present in the current agent.
3. **Testing Capability:** The agent must be capable of developing comprehensive tests for the next-generation agent based on the specifications.
4. **Self-Testing & Feedback:** The agent should be able to execute these tests and receive feedback on its own performance.
5. **Multi-File Project Structure:** If the next-generation agent consists of multiple source files, it should be organized in a dedicated `agentN` folder, which will serve as the base directory.
6. **Test-Gated Evolution:** The evolution to the next-generation agent should only occur after it has successfully passed all defined tests.
7. **Reference vs. Usage:** During the development of the next-generation agent, the source code of previous agents can be referenced for guidance but does not strictly need to be reused.
8. **Phased Development:** You will have to separate this dev to multiple small phases before starting bold development. Due to limited context from the local LLM setup, phase design and development are necessary; phases could be coded into the role message or a static part of memory. Do not fear to remove and rebuild modules if they are shitty and hard to continue maintaining.

### **Current Agent Features (Agent0 / Base Agent)**

**1. Core Role & Purpose:**
The agent functions as a coding agent designed to build llama server-based Python coding agents. Its primary tasks involve modifying code, explaining code, and suggesting commands, and everything related to it.

The role message includes the following important instructions:
- If the command exit code is 0 and there are no errors in Stderr, consider the command successful, proceed to the next step or provide the final answer.
- If there are no commands to execute, the "commands" array MUST be empty.
- You need to simply acknowledge the LAST command executed in a sequence, even if you did not expect it returns to you anything.
- Agent0 source is in agent0.py, single file.

**2. Memory Management System (`MemoryManager` class):**
- **JSON-Based Storage:** Maintains conversation history in a `memory.json` file as a JSON array of messages with `role` and `content` fields (e.g., `{"role": "system/user/assistant", "content": "..."}`).
- **Format Conversion:** Automatically converts old format memory (dict with `system`, `user`, `assistant`, `file` keys) to the new format (list of messages).
- **System Context Enforcement:** Automatically ensures the role definition (system message) is the first message in the memory via the `init_context()` method.
- **Execution Output Discard:** Provides a `discard_execution_outputs()` method to remove execution output messages from memory (messages with role 'user' and content starting with '--- Execution Output for LLM') when the final response is reached (no commands to execute).

**3. Interactive Chat Commands (`ChatAgent` class):**
The agent supports an interactive terminal interface with the following slash commands:
- `/help`: Displays available commands.
- `/save`: Saves the current memory to the JSON file.
- `/load`: Loads memory from the JSON file.
- `/temp <float>`: Sets the LLM's temperature parameter (valid range: 0.0 to 2.0). The current temperature is displayed in the user prompt (e.g., `[U|0.3]`).
- `/dp`: Toggles debug prints (prints the messages built for API request if debug prints are enabled).

*(Note: `/read <filename>` and `/exec <command>` commands have been removed. Command execution is now handled exclusively via the auto-execution loop based on JSON grammar response.)*

**4. LLM Integration & Auto-Execution Loop:**
- **API Connection:** Connects to an LLM API (default: `http://localhost:8080/v1/chat/completions`) using the OpenAI-compatible chat completions format with `response_format: {"type": "json_schema", ...}`.
- **JSON Grammar Response:** The LLM is instructed to respond with a valid JSON object following this schema:
  ```json
  {
    "message": "Your explanation, final answer, or conversational text. If there are no commands to execute, this contains the final answer.",
    "commands": [
      "command1",
      "command2"
    ]
  }
  ```
- **Auto-Execution Loop:** When the LLM provides a response containing executable commands in the `commands` array, the agent executes them, captures the output (stdout/stderr/exit code), and feeds that output back to the LLM in the next iteration (up to a maximum of 1000 iterations).
- **Message Building:** Uses the `memory_manager.memory_json` directly (a JSON array of messages) for the API request. Execution output messages are added as 'user' messages starting with '--- Execution Output for LLM (Iteration X) ---', and are discarded via `discard_execution_outputs()` when the final response (no commands) is reached.

**5. Command Execution & Validation:**
- **Command Extraction:** Commands are extracted directly from the `commands` array in the JSON grammar response. XML guard rails and ```bash ``` language filtering have been removed in favor of the structured JSON `commands` array.
- **Command Timeout:** The auto-execution loop enforces a 900-second timeout in `subprocess.run()` to prevent hanging or infinite-looping shell commands. (Note: The error message for timeout incorrectly states "timed out after 60 seconds", but the actual `timeout` parameter is set to 900).

**6. Security & Safety Features:**
- **Command Timeout:** Enforces a 900-second timeout in `subprocess.run()` to prevent hanging or infinite-looping shell commands.
- **No Re-execution Instruction:** After successfully executing commands (Exit Code: 0, no Stderr errors), the role message instructs the LLM to proceed to the next step or provide the final answer, ensuring it does not re-execute the same commands.

**7. Existing Issues & Pending Improvements:**
- Sometimes the server may return an empty response, which might be caused by conversations that are too long. Retyping the request sometimes works around this issue.

### **Agent1 Features (Pending / Future Evolution)**

**Core Philosophy:**
The most essential part of Agent1 is making it able to be tested, and building a feedback loop for itself to mature. Agent1 should be a multiple-file project.

**Core Features (Pending):**
- **Self-Reading & Multi-File Support:** Agent1 should be a multiple-file project. It should support self-reading without attaching source code in the system prompt as a "Current Agent Source Code" title message, and support multi-file project structures natively.
- **Source Code Change Detection:** Detect if the source code has changed and update the source code reflection in the system prompt accordingly.
- **Valid Command Checker:** Filter and validate commands before execution to ensure they are non-empty and valid strings.
- **Subagent Generation:** Not yet implemented. Future agent should be capable of generating a subagent.
- **Conversation Compaction & Forgetting:** Not yet implemented. Advanced conversation compaction and forgetting upon user demand are not yet implemented.
- **Role Mutation & Specialization:** Not yet implemented. Future agent should be able to modify its role to fork and mutate itself into a specialized agent (e.g., a C++ LeetCode solver) if requested by the user.

**User Interface (Pending / Partially Implemented):**
- **Improved Prompt Editing:** Not yet implemented (currently uses standard `input()`). Future version should allow moving the cursor within the prompt and inserting text in the middle, avoiding issues like generating escape characters (e.g., `^[[C`).
- **Token & Rate Display:** *Implemented.* The number of tokens used (`completion_tokens`) and total tokens (`total_tokens`) are displayed per request return, highlighted in red color (e.g., `[A|prompt_tokens|completion_tokens|total_tokens_accumulator]`).
- **`/usage` Command (Global Statistics):** Not yet implemented. Future agent should add a `/usage` feature that can print global statistics, such as how much tokens have been used, request counts, and other relevant usage metrics, to the UI section.
